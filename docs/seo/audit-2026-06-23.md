## 2. 합의된 CSR 제약의 측정된 비용 (재검토용 데이터, 강제 아님)

기존 합의: "구글봇은 JS를 렌더링하므로 CSR+helmet으로 충분, 네이버·SSR은 범위 외." 이 전제는 **구글 검색만 본다면 타당하다.** 다만 이번 감사가 정량화한 **그 결정의 실제 비용**은 다음과 같다. 결정을 바꾸라는 게 아니라, 비용을 알고 유지하자는 취지다.

- **AI 검색 가시성 ≈ 0 (GEO 12/100).** GPTBot·ClaudeBot·PerplexityBot·Google-Extended·NaverBot은 JS를 실행하지 않으므로 **전원 빈 셸(503바이트)만 수신.** 339개 제품의 영양·성분 데이터(높은 인용가치)가 AI 답변에 전혀 노출되지 않는다. 2026년 기준 다이어트/영양 질의의 상당수가 AI Overviews·ChatGPT·Perplexity로 이동 중 — 이 채널이 통째로 닫혀 있다. (기존 계획 수립 시점엔 "구글 검색"만 고려, AI 채널은 미반영.)
- **SNS 공유 미리보기**: 제품별 OG가 helmet(JS)이라 카카오·페북봇엔 제네릭 이미지로만 노출. (기존 계획도 한계로 인지함.)
- **소프트404·중복URL**이 CSR 셸 반환에서 파생(아래 §3-2, §3-8).

**판단 가이드:** 구글 검색 유입만이 목표라면 현 전략 유지 가능. **AI 검색·SNS·네이버 중 하나라도 전략 목표가 되면**, 빌드타임 프리렌더(SSG, `vite-react-ssg` 등)가 가장 큰 레버리지다. 339개는 정적 생성에 충분히 작은 규모이고, 산출물(helmet 메타·JSON-LD)은 그대로 재사용된다.

## 3. 발견 사항 (우선순위별, 신규 ⭐ 표시)

### 🔴 CRITICAL

**C1. CSR 구조 — 모든 콘텐츠/메타/스키마가 JS 뒤에 존재** `[전 에이전트]`
서버 HTML 1.6KB 빈 셸, 전 라우트 동일 ETag. helmet 산출물(canonical·메타·JSON-LD)은 렌더(2.6~5.9초) 후에만 존재. → 구글봇은 2차 렌더링으로 일부 인식(합의 전제), 그 외 크롤러는 못 봄. *처방: §2 참조(범위 외 결정).*

**C2. 소프트 404 — 무효 경로가 HTTP 200** `[기술·사이트맵]`
`/없는경로`·`/sitemap.xml`·`/category/protein-shake`(영문 슬러그) 전부 200+셸. CSR 제약상 서버 404가 어려우면 **최소 Phase 3.2의 noindex 메타를 모든 무효 경로에 일괄 적용**(현재 무효 제품 상세만 적용 추정). 여력 시 `vercel.json`으로 알려진 무효 패턴에 실제 404.
→ 검증: `curl -o /dev/null -w "%{http_code}" .../없는경로` → 404 또는 렌더 DOM에 noindex.

**C3. ⭐ YMYL 신뢰 — 작성자·면책·출처 모순** `[콘텐츠]` (어느 Phase에도 없던 신규)
건강 의사결정에 쓰이는 YMYL 사이트:
- **작성자/편집 주체·자격 표기 없음**, **사업자등록번호 누락** — 제휴수익(쿠팡 링크) 사이트는 전자상거래법상 사업자 정보 표기 의무. 법적 노출 + 구글 QRG(2025-09) YMYL E-E-A-T 하드 실패 조건.
- **의료·영양 면책 문구 0건** (제품 상세가 단백질 품질 등급 등 영양 판단을 렌더하면서).
- **`/about`은 "식약처 공시 기준"이라 명시하나 실제 58.1%(197/339)가 쿠팡·브랜드몰 출처.** 검증 가능한 모순 → 신뢰도·AI 인용가치 동시 훼손.
→ 처방: 운영주체/사업자번호 푸터, 사이트 면책 문구, **제품별 출처 배지(식약처/브랜드 제공)**, `/about` 문구를 실제와 일치하게 정정.

### 🟠 HIGH

**H1. ⭐ 전 페이지 401/404 콘솔 에러** `[기술·비주얼]` (신규)
홈 1건, `/product/190` 2건, `/list` 5건(401×1, 404×4) — Supabase 호출 일부 실패. **WRS가 데이터 실패 상태로 렌더→빈 콘텐츠 색인 위험** + INP 악화. → 실패 엔드포인트/anon키 권한 audit, 에러 바운더리·AbortController. *(C1보다 우선 처리 필요: SSR을 도입해도 데이터가 실패하면 빈 페이지가 색인됨.)*

**H2. ⭐ 매 페이지 2.4MB Supabase 전체 테이블 fetch** `[성능]` (신규)
`GET /foods?select=...` 가 339행 전체(2,519,401 bytes)를 **모든 라우트에서** 로드. LCP 임계경로를 점유. → `limit/offset` 페이지네이션 + 목록용 컬럼만 select(약 94%↓), 상세에서만 전체 영양 조회.

**H3. h1 태그 전 페이지 0개** `[비주얼·기술]` (Phase 4.6 미완)
히어로·제품명·목록 제목이 `<p>/<div>`. → `<h1>`로 래핑.

**H4. ⭐ 제품 URL 중복** `[기술]` (신규)
`/product/190`(숫자) 와 `/product/<한글>-190`(슬러그) **둘 다 200** — 339개 → 최대 678 URL. → 숫자→슬러그 301, 사이트맵·내부링크는 슬러그만.

**H5. Product JSON-LD 결함** `[기술·스키마]`
`image`가 base64 데이터URI(구글 크롤 불가→리치결과 무효), `NutritionInformation`·`sku` 누락. → image를 HTTPS URL로(Supabase Storage `storage_image_path` 활용), 영양 스키마 추가(§5 산출물). *영양 스키마는 이 사이트 핵심 가치라 가장 차별화되는 항목.*

**H6. ⭐ 엔트리 번들 426KB + Gmarket Sans 폰트 파일 누락** `[성능]` (신규)
`/fonts/GmarketSansTTF*.ttf` 요청이 1.6KB 셸 반환 = 배포 누락(시스템폰트 폴백 중). 엔트리 번들 426KB(브로틀리 128KB)는 라우트 분할에도 불구 큼. → 폰트 WOFF2로 `public/fonts/` 정확 배포, 번들 분석(`vite-bundle-visualizer`)으로 Supabase SDK 등 분리.

### 5-2. 카테고리/제품 URL 패턴 (사이트맵 생성 시 필수)
- 카테고리: `/category/<한글슬러그>` (코드 아님). 예: `shake→/category/셰이크`, `protein_drink→/category/단백질-음료`, `chicken_breast→/category/닭가슴살`, `ice_cream→/category/아이스크림`, `snack_sweets→/category/과자-초콜릿-젤리`, `zero_drink→/category/제로-음료`. (energy_bar/cereal/processed_meat/noodle 슬러그는 프런트 번들의 매핑 객체에서 확인 필요.)
- 제품: `/product/<한글이름-슬러그>-<id>` (숫자 단독 아님). canonical 실측 예: `/product/하림-닭가슴살-블랙페퍼-5`.

## 6. 권장 실행 순서 (기존 Phase + 신규 통합)
